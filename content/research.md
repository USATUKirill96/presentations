# Research: AI-Assisted Development — практики продуктивной разработки

> Составлено: 24 марта 2026
> Охват: универсальные паттерны, применимые к Claude Code, Cursor, GitHub Copilot и другим инструментам
> Источники: академические исследования, официальная документация, инженерные блоги, отчёты DORA и METR

---

## Ключевые инсайты (топ-5 главных мыслей для презентации)

1. **От prompt engineering к context engineering.** Качество вывода AI-агента определяется не одним промптом, а всей экосистемой контекста: файлы проекта, правила, история диалога, подключённые инструменты. Управление этим контекстом — ключевой навык 2025-2026.

2. **Plan → Implement — универсальный паттерн.** Разделение фазы исследования и фазы реализации работает в любом инструменте. Markdown-файл плана — мост между намерением человека и исполнением агента, который переживает сбросы контекста и смену инструментов.

3. **Роль разработчика сдвигается от writer к orchestrator.** Самые ценные навыки теперь — декомпозиция задач, курирование контекста, ревью результата и интеграция. Параллельные AI-сессии позволяют вести несколько задач одновременно, как оркестратор.

4. **AI — усилитель, не замена.** Отчёт DORA 2025: AI «усиливает существующие сильные и слабые стороны организации». Команды с хорошими инженерными практиками (тесты, ревью, CI/CD) получают мультипликативную отдачу. Команды без них — мультипликативный технический долг.

5. **Все инструменты конвергируют к одним и тем же паттернам.** Персистентный контекст (CLAUDE.md / .cursorrules / copilot-instructions.md), подключение внешних сервисов (MCP), кастомные команды, план-файлы — каждый инструмент независимо пришёл к одним и тем же решениям. Инвестиция в универсальные практики окупается при любой смене инструментов.

## A. Возможности (часто игнорируемые)

### Персистентный контекст проекта

**Что это:** Markdown-файл в репозитории, который AI-агент автоматически читает в начале каждой сессии. Каждый инструмент назвал его по-своему, но функция одна и та же: «память проекта», которая не теряется между сессиями.

**Как это реализовано в разных инструментах:**

| Файл | Инструмент | Особенности |
|------|-----------|-------------|
| `CLAUDE.md` | Claude Code | Иерархия: глобальный (~/.claude/), проектный, поддиректории |
| `.cursor/rules/*.mdc` | Cursor | Гранулярные правила с glob-скоупами для разных частей проекта |
| `.github/copilot-instructions.md` | GitHub Copilot | Поддерживает подпапку `.github/instructions/` для скоупированных правил |
| `AGENTS.md` | Кросс-инструментальный стандарт | Поддерживается Codex CLI, Cursor, Claude Code, Aider, OpenHands |
| `.windsurfrules` | Windsurf | Аналогичная функция, другой формат |
| `GEMINI.md` | Gemini CLI | Иерархическая конфигурация |

**Пример содержимого (универсальный):**
```markdown
# Code Style
- ES modules (import/export), не CommonJS
- Деструктуризация импортов где возможно

# Workflow
- Typecheck после любых изменений кода
- Запускать отдельные тесты, не весь набор

# Architecture
- State management: Zustand (см. src/stores/)
- API conventions: @docs/api-conventions.md

# Environment
- Используем bun, не npm
- bun test перед каждым коммитом
```

**Что включать / не включать:**

| Включать | Не включать |
|----------|-------------|
| Команды сборки/тестирования, которые AI не угадает | То, что AI и так видит в коде |
| Code style, отличающийся от стандартного | Общеизвестные конвенции языка |
| Архитектурные решения, специфичные для проекта | Файл-по-файл описание кодовой базы |
| Gotchas и неочевидные поведения | Самоочевидное вроде «пиши чистый код» |

**Подход «один файл — все инструменты»:** Некоторые команды ведут один канонический markdown-файл с инструкциями и симлинкуют его в нужные места (CLAUDE.md, .cursorrules, copilot-instructions.md). Это обеспечивает консистентное поведение при смене инструмента.

**Исследование (arXiv:2602.14690):** Анализ реальных репозиториев показал, что контекстные файлы — «часто единственный механизм конфигурации» AI-агентов. Большинство разработчиков используют только 1-2 таких файла. Продвинутые возможности (скиллы, субагенты, хуки) остаются недоиспользованными.

**Главная ловушка:** Раздутый контекстный файл контрпродуктивен — важные правила теряются в шуме. Для каждой строки задавайте вопрос: «Если убрать это — AI сделает ошибку?» Если нет — удалить.

### Хуки и автоматизация

**Что это:** Скрипты, которые автоматически выполняются на события жизненного цикла агента. Критическое отличие от контекстных файлов: хуки — детерминированные. Контекстный файл — рекомендация, которую AI может проигнорировать. Хук — принуждение, которое выполняется всегда.

**Реализация в инструментах:**

| Инструмент | Механизм | Формат |
|-----------|---------|--------|
| Claude Code | `hooks` в settings.json | JSON с matcher + shell command |
| Cursor | Rules с auto-attach по glob | MDC-файлы с фронтматтером |
| GitHub Copilot | GitHub Actions + custom instructions | YAML workflows |
| Aider | Pre-commit hooks + `.aider.conf.yml` | Стандартные git hooks |

**Типичные сценарии (универсальные):**
- Автоформатирование файлов после редактирования (prettier, black, gofmt)
- Блокировка изменений в чувствительных файлах (.env, credentials)
- Десктопная нотификация когда агент ждёт ввода
- Автозапуск линтера/тайпчекера после изменений
- Логирование всех действий агента

**Пример (Claude Code):**
```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{ "type": "command", "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write" }]
      }
    ]
  }
}
```

**Пример (Cursor — auto-attach правило):**
```markdown
---
description: Форматирование TypeScript файлов
globs: ["**/*.ts", "**/*.tsx"]
alwaysApply: true
---
After editing any TypeScript file, run `npx prettier --write` on it.
Never modify .env files.
```

**Ключевой принцип:** Правила, которые должны соблюдаться *всегда* — переносите из контекстного файла в хуки/автоматизацию. Контекстный файл — для рекомендаций, хуки — для принуждения.

### Подключение внешних инструментов (MCP)

**Что это:** Model Context Protocol (MCP) — открытый стандарт для подключения AI-агентов к внешним инструментам и источникам данных. Часто описывается как «USB-C для AI» — универсальный коннектор, который превращает M×N интеграций в M+N.

**Статус (март 2026):** 97 млн скачиваний SDK в месяц, 10 000+ активных серверов. Поддерживается Claude Code, Cursor, VS Code (Copilot), Windsurf, Cline, и другими.

**Архитектура:**
- **Hosts:** Приложения с AI (Claude Desktop, Cursor, VS Code)
- **Clients:** Коннекторы внутри host, управляющие подключениями
- **Servers:** Предоставляют возможности через три примитива:
  - **Tools** — функции, которые AI может вызывать (запросы к БД, создание тикетов)
  - **Resources** — данные для чтения (схемы БД, документация)
  - **Prompts** — предопределённые workflow и шаблоны

**Популярные MCP-серверы для разработки:**

| Сервер | Что даёт |
|--------|----------|
| GitHub MCP | Работа с репозиториями, PR, issues, CI/CD |
| Playwright MCP | Браузерная автоматизация, E2E тесты |
| Sentry MCP | Доступ к ошибкам и мониторингу |
| PostgreSQL MCP | Запросы к базе данных |
| Jira / Linear MCP | Задачи и проекты |
| Figma MCP | Доступ к дизайн-файлам |

**Реальные сценарии:**
- «Реализуй фичу из JIRA-тикета ENG-4521 и создай PR на GitHub»
- «Проверь Sentry, найди ошибки связанные с auth flow»
- «Найди в PostgreSQL 10 пользователей, которые использовали feature X»

**Безопасность:** MCP-серверы от третьих лиц — потенциальный вектор prompt injection. Использовать только из доверенных источников. Управление разрешениями — «MUST, не SHOULD».

**Почему важно:** MCP позволяет AI-агенту работать в едином рабочем пространстве — без переключения между GitHub, Jira, мониторингом и кодом. Это инструмент-агностичный стандарт: один раз настроил MCP-сервер — работает в любом поддерживающем AI-инструменте.

### Кастомные команды и скиллы

**Что это:** Повторяемые рабочие процессы, оформленные как команды. Превращают 20-шаговый workflow в одну команду. Новый разработчик видит набор команд проекта и сразу понимает, как здесь работают.

**Реализация в разных инструментах:**

| Инструмент | Механизм | Расположение |
|-----------|---------|-------------|
| Claude Code | Slash commands + Skills | `.claude/commands/`, `.claude/skills/` |
| Cursor | Notepads + Custom Modes | Notepads для контекста, Custom Modes для поведения |
| Copilot | Reusable prompts + chat participants | `.github/copilot-instructions.md`, расширения VS Code |
| Aider | Custom prompts в `.aider.conf.yml` | Конфигурационный файл |

**Пример: автоматизация issue → PR (универсальный):**
```markdown
# fix-issue
Проанализируй и исправь issue: $ARGUMENTS.

1. Получи детали issue (gh issue view / jira view)
2. Разберись в проблеме
3. Найди релевантные файлы в кодовой базе
4. Реализуй исправление
5. Напиши и запусти тесты
6. Проверь lint и type checks
7. Создай осмысленный коммит и открой PR
```

**Пример: code review перед пушем:**
```markdown
# review
Проверь изменения в текущей ветке перед пушем.

1. Посмотри diff с main
2. Проверь: безопасность, покрытие тестами, breaking changes, TODO в коде
3. Запусти тест-сьют
4. Выведи структурированный отчёт: OK / Warning / Must fix
5. Если критические проблемы — НЕ пушить, сначала исправить
```

**Пример: создание компонента по шаблону проекта:**
```markdown
# component
Создай новый React-компонент $ARGUMENTS по конвенциям проекта.

1. Создай $ARGUMENTS.tsx с TypeScript props interface
2. Создай $ARGUMENTS.test.tsx с базовым тестом
3. Создай index.ts с re-export
4. Добавь story в $ARGUMENTS.stories.tsx
5. Следуй паттернам из существующего Button компонента как референс
```

**Notepads в Cursor** — отдельная сущность: переиспользуемые блоки контекста, которые можно подключать к разным запросам через @-упоминание. Это не команды, а «контекст по требованию»: архитектурные решения, стайлгайды, описания API.

**Почему важно:** Команда с 5-10 задокументированными скиллами/командами сокращает onboarding. Скиллы — живая документация: они не устаревают, потому что ссылаются на реальные файлы проекта.

### Параллельные AI-сессии

**Что это:** Запуск нескольких независимых AI-сессий одновременно, каждая работает над своей задачей в изолированном окружении.

**Ключевой enabler: Git worktrees.** Git worktree позволяет иметь несколько рабочих директорий с разными ветками из одного репозитория. Каждая AI-сессия работает в своём worktree — никаких конфликтов файлов.

**Реализация:**

| Инструмент | Механизм |
|-----------|---------|
| Claude Code | `claude --worktree feature-name` (встроенная поддержка) |
| Cursor | Несколько окон в разных worktrees |
| Copilot Coding Agent | Автономные PR из issues (каждый в своей ветке) |
| Aider | Отдельные сессии в разных worktrees |

**Паттерны:**

1. **Writer/Reviewer:** Сессия A пишет код. Сессия B (свежий контекст) ревьюит результат. Свежий контекст ловит проблемы, к которым «писатель» уже привык.

2. **Writer/Tester:** Сессия A пишет тесты. Сессия B пишет код, который проходит эти тесты. Естественное TDD через разделение ролей.

3. **Параллельная разработка:** Несколько задач из бэклога — каждая в своём worktree. Человек — оркестратор, а не исполнитель.

**Реальный пример — Lightspark:** CTO компании описывает запуск «множества параллельных AI-сессий, каждая в изолированном worktree». Люди «проектируют workflow», AI исполняет. Используют «структурированные многораундные дебаты», где агенты ревьюят работу друг друга.

**Критические требования:**
- **Чёткие границы задач.** Параллельные агенты работают только если задачи касаются разных частей кодовой базы. Пересекающиеся файлы — merge hell.
- **Изоляция не абсолютна.** Worktrees НЕ изолируют базу данных, Docker, кеш-директории.
- **Дисковое пространство:** 2 GB кодовая база × 5 сессий = ~10 GB.
- **Человеческая оркестрация обязательна.** Роль человека — декомпозировать работу, распределить задачи, интегрировать результаты.

### Режим планирования (Plan Mode)

**Что это:** Режим, в котором AI только читает и анализирует код, но не вносит изменений. Отделение фазы исследования от фазы реализации.

**Реализация:**

| Инструмент | Как включить |
|-----------|-------------|
| Claude Code | Plan Mode (Shift+Tab), автоматический think → plan → code loop |
| Cursor | Plan mode в Composer (модель-мыслитель + модель-кодер) |
| Copilot | Ask mode (Chat) → Edit mode (Copilot Edits) — разделение вручную |
| Aider | `/architect` mode для планирования |

**Рекомендованный 4-фазный workflow (универсальный):**

1. **Explore:** «Прочитай /src/auth и пойми, как мы обрабатываем сессии»
2. **Plan:** «Я хочу добавить Google OAuth. Какие файлы нужно изменить? Составь план.»
3. **Implement:** «Реализуй OAuth flow из твоего плана. Напиши тесты, запусти, исправь ошибки»
4. **Commit:** «Закоммить с осмысленным сообщением и открой PR»

**Markdown план-файл (tool-agnostic best practice):**
Вместо встроенного планирования (которое теряется при сбросе контекста) — записывать план в markdown-файл. Этот файл переживает сбросы контекста, смену инструментов и передачу задачи другому разработчику.

Структура план-файла:
- Нумерованная таблица задач (задача, статус, приоритет, комментарии)
- Разбивка по дизайну/архитектуре
- Лог ошибок и решений
- TODO-секция

Нумерация позволяет давать точные команды: «Сделай 1.1 — 1.3, пропусти 1.4.»

**Когда пропускать планирование:** Задача маленькая и очевидная — исправить опечатку, добавить импорт, переименовать переменную.

### Контроль разрешений

**Что это:** Система управления тем, что AI-агент может делать без подтверждения. Баланс между продуктивностью (меньше прерываний) и безопасностью (контроль над опасными действиями).

**Типичные уровни (реализуются по-разному, но паттерн универсален):**

| Уровень | Поведение | Когда |
|---------|-----------|-------|
| **Restrictive** | Спрашивает перед каждым действием | Незнакомый проект, начало работы |
| **Moderate** | Авто-чтение/запись файлов, shell — с подтверждением | Доверенный проект, активная разработка |
| **Read-only** | Только чтение и анализ | Аудит, понимание кодовой базы |
| **Full auto** | Всё без подтверждений | Только sandbox/CI без интернета |

**Deny rules — важнее allow rules.** В любом инструменте первым делом заблокируйте опасные операции:
```
git push --force
git push origin main
rm -rf
git reset --hard
```

**Принцип:** Автоматизируйте рутину (чтение, запись, запуск тестов) → спрашивайте перед опасным (удаление, force push, деплой).

## B. Практики и паттерны

### Контекст-менеджмент

**Контекстное окно — самый дорогой ресурс.** Производительность AI-агента деградирует по мере заполнения контекста нерелевантной информацией. Управление контекстом — не nice-to-have, а core skill.

**Универсальные стратегии:**

**1. Компакция / сброс.** Когда разговор становится длинным — суммировать предыдущую историю, сохранив архитектурные решения и нерешённые баги. В Claude Code это `/compact`, в Cursor — новый Composer, в Copilot — новый чат. Принцип один: регулярно «освежать» контекст.

**2. Структурированные заметки (внешняя память).** Агенты записывают прогресс и решения во внешние файлы (план, TODO, scratchpad). Это позволяет отслеживать прогресс даже после полного сброса контекста. Паттерн «внешний мозг».

**3. Just-in-time retrieval.** Вместо предзагрузки всей кодовой базы — подгружать информацию по мере необходимости через поиск по файлам и коду. Лучшие агенты навигируют кодовую базу «по требованию», а не пытаются прочитать всё разом.

**4. Изоляция через субагенты.** Делегировать фокусные подзадачи субагентам, которые возвращают только сжатое резюме. Это предотвращает «загрязнение» основного контекста деталями подзадач.

**Когда сбрасывать контекст:**
- Между несвязанными задачами
- После двух неудачных попыток исправить одно и то же
- Когда AI начинает «забывать» ранние инструкции или делать странные ошибки

**Ключевая мысль (Anthropic):** Сдвиг от «prompt engineering» (одиночный запрос) к «context engineering» (управление всей информационной экосистемой) — определяющий навык 2025-2026.

### Итеративная работа с агентом

**Антипаттерн:** Монолитный промпт — «реализуй всю систему аутентификации с OAuth, MFA и сбросом пароля». Это заставляет AI «принимать компромиссы, о которых вы не просили».

**Правильный подход — декомпозиция:**

1. **Определить scope.** Не «исследуй нашу систему аутентификации», а «найди, как обрабатывается token refresh в src/auth/».

2. **Итеративный цикл.** Генерация → ревью → уточнение конкретных аспектов → повторная генерация. Каждая итерация — с фокусной целью.

3. **Интервью перед большой фичей.** Попросить AI задать вопросы о реализации, UI/UX, edge cases и trade-offs *перед* тем, как начать кодить. Записать результат в спек-файл. Новая сессия — для реализации.

4. **Подача контекста:**
   - Ссылки на файлы: `@src/auth/login.ts` (Cursor, Claude Code)
   - `#file` в Copilot
   - Скриншоты напрямую в промпт
   - Pipe данных: `cat error.log | claude` или вставка в чат
   - URL документации

**Четыре навыка промптинга (2026):**
- **Instructing** — что сделать
- **Constraining** — что НЕ делать, ограничения
- **Providing context** — дать релевантную информацию
- **Evaluating** — оценить и уточнить результат

### Ревью кода агентом

**Writer/Reviewer pattern — самый высокоROI-ный паттерн AI-разработки.**

Принцип: свежая AI-сессия без знания о компромиссах реализации ловит проблемы, к которым «писатель» уже привык.

**Трёхуровневая архитектура ревью:**

1. **IDE-уровень** (Cursor, Copilot inline): Realtime-ревью по мере написания. Ловит синтаксические проблемы, очевидные баги, стилевые нарушения.

2. **PR-автоматизация** (CodeRabbit, Cursor Bugbot, Copilot auto-review): Триггерится на pull request. Ревьюит дифф на логические ошибки, безопасность, производительность. Сортирует: «Must-fix» / «Worth considering».

3. **Архитектурный уровень** (Claude Code / Cursor Agent с полным контекстом репо): Глубокий ревью дизайн-решений, кросс-модульных зависимостей, системных последствий.

**Практический подход:**
```text
Prompt для ревьюера:
"Проревьюй реализацию в src/middleware/rateLimiter.ts.
Ищи: edge cases, race conditions, соответствие нашим паттернам.
Дай конкретные ссылки на строки и предложи исправления."
```

**Ключевое правило:** Никогда не автоматизируйте 100% ревью. AI отлично находит паттерн-based проблемы (null checks, error handling, naming), но не может надёжно оценить бизнес-логику, архитектурную уместность и compliance.

### TDD с AI

**Главная проблема:** AI-агенты по умолчанию оптимизируют на *завершение*, а TDD требует *сдержанности*. Без явного принуждения AI напишет имплементацию перед тестами.

**Почему это сложно:** Когда просишь «build feature X with TDD», AI часто пишет и тесты, и реализацию одновременно, пропуская Red-фазу. Тесты получаются «подогнанные» под имплементацию, а не наоборот.

**Рабочие стратегии (универсальные):**

**1. Явное разделение ролей:**
```text
# Шаг 1 (Red)
"Ты — тестер. Напиши failing тесты для rate limiter:
- 100 запросов в минуту на пользователя
- 429 после превышения лимита
- Сброс счётчика через 60 секунд
НЕ ПИШИ реализацию. Покажи output падающих тестов."

# Шаг 2 (Green)
"Теперь реализуй RateLimiter, чтобы пройти все тесты.
Минимальная реализация."

# Шаг 3 (Refactor)
"Тесты проходят. Теперь рефактори для чистоты и производительности,
не ломая тесты."
```

**2. Принуждение через контекстный файл:**
```markdown
# TDD Rules
- НИКОГДА не пиши имплементацию перед тестами
- Используй red-green-refactor цикл явно
- Тесты должны падать перед имплементацией (покажи вывод)
- По одному тесту за раз
```

**3. Specification-Driven Development (SDD) как альтернатива.**
Написать детальную спецификацию поведения → AI генерирует и тесты, и реализацию из спецификации. Даёт AI больше структурированного контекста, чем чистый TDD, сохраняя принцип «определи поведение перед реализацией».

**4. Отдельные сессии для Red/Green.** Red-сессия пишет только тесты. Green-сессия пишет только имплементацию. Физическое разделение предотвращает «дрифт».

**Где AI особенно полезен в тестировании:** Генерация тест-кейсов. AI отлично находит edge cases, о которых человек не подумал. Используйте эту силу: попросите AI «сгенерировать 10 edge case тестов для этой функции» — это часто самый ценный use case.

### Git workflow

**Паттерны работы с git через AI-агенты:**

**1. Feature branch per session:**
```bash
# Claude Code
claude --worktree feature-name

# Cursor / Copilot
git worktree add ../feature-name -b feature/name
# Открыть в новом окне IDE
```

**2. Non-interactive в CI:**
```bash
# Claude Code
claude -p "Review this PR diff for security issues" --output-format json

# Copilot Coding Agent
# Назначить issue на @copilot → автоматический PR
```

**3. Batch-операции:**
```bash
for file in $(cat files.txt); do
  claude -p "Migrate $file from Python 2 to Python 3. Return OK or FAIL." \
    --allowedTools "Edit,Bash(git commit *)"
done
```

**4. Исследование истории:**
```text
"Посмотри git history ExecutionFactory и расскажи,
как эволюционировал её API и почему"
```

**Важные deny rules (универсально):**
```
git push --force
git push origin main
git reset --hard
rm -rf
```

**Коммит-дисциплина:** Коммитить после каждого успешного шага. Это даёт точки отката, если AI поведёт код в неправильном направлении. Без частых коммитов «вы не сможете откатить» — и придётся начинать заново.

## C. Антипаттерны

### 1. Слишком общие промпты

**Антипаттерн:**
```
"fix the login bug"
"make the dashboard look better"
```

**Правильно:**
```
"Пользователи сообщают, что логин падает после таймаута сессии.
Проверь auth flow в src/auth/, особенно token refresh.
Напиши failing тест, воспроизводящий проблему, затем исправь."
```

AI «изобретает модель данных», когда не получает контекст. Чем больше ограничений (технология, паттерны, edge cases), тем детерминированнее результат.

---

### 2. «Kitchen sink» сессии

**Антипаттерн:** Начал с одной задачи → спросил несвязанное → вернулся к первой. Контекст полон шума.

**Симптомы:** AI «забывает» ранние инструкции, делает странные ошибки, противоречит собственным решениям.

**Правило:** Сброс контекста между несвязанными задачами. После двух неудачных попыток — сброс и лучший промпт.

---

### 3. «Accept All» — слепое принятие

**Антипаттерн:** AI написал правдоподобно выглядящий код — принять без проверки. Самая частая ошибка по всем источникам.

**Последствие:** Необработанные edge cases, race conditions, баги в production. Код, который разработчик не понимает.

**Правило:** Всегда давать AI средства верификации — тесты, скриншоты, expected output. Если нельзя верифицировать — не шипать. Относитесь к выводу AI как к черновику, не к ответу.

---

### 4. Раздутый контекстный файл

**Антипаттерн:** CLAUDE.md / .cursorrules на 500 строк с каждой конвенцией.

**Последствие:** Важные правила теряются. AI их не игнорирует намеренно — контекст просто переполнен.

**Правило:** Тест для каждой строки: «Без этого AI ошибётся?» Если нет — удалить или перенести в хук/автоматизацию.

---

### 5. Монолитный промпт вместо итераций

**Антипаттерн:** «implement the entire user authentication system with OAuth, MFA, and password reset»

**Последствие:** AI принимает архитектурные решения, с которыми вы бы не согласились. «Компромиссы, о которых вы не просили.»

**Правило:** Explore → Plan → Implement. Для больших фич: интервью → spec в файл → новая сессия для реализации.

---

### 6. Бесконечное исследование без scope

**Антипаттерн:** «investigate our authentication system» без ограничений.

**Последствие:** AI читает сотни файлов, переполняет контекст, сессия деградирует.

**Правило:** Скоупить исследования узко. Делегировать субагентам/отдельным сессиям.

---

### 7. Архитектурные решения на откуп AI

**Антипаттерн:** Позволить AI неявно принимать архитектурные решения при реализации.

**Последствие:** Код, который «вы не сможете дебажить, когда он сломается». Архитектурный drift.

**Правило:** Архитектурные решения — всегда человеческие и явные. AI реализует, человек проектирует.

---

### 8. Никогда не кодить руками

**Антипаттерн:** Использовать AI для всего, включая тривиальные изменения.

**Последствие:** Overhead промпта для переименования переменной или добавления лога превышает время набора руками.

**Правило:** Некоторые задачи быстрее без AI. Промпт-overhead реален. Используйте AI там, где он даёт кратный выигрыш, не повсеместно.

---

### 9. Нет version control дисциплины

**Антипаттерн:** Не коммитить после каждого шага.

**Последствие:** Когда AI ведёт код в неправильном направлении — нет точки отката. Приходится начинать заново.

**Правило:** Коммит после каждого успешного шага. Это страховка.

---

**Сценарий: Миграция большого количества файлов**

```bash
for file in $(cat files.txt); do
  [ai-tool] -p "Migrate $file from Python 2 to Python 3. Return OK or FAIL."

# 3. Тестируем на 2-3 файлах, потом полный прогон
```

**Сценарий: Рефакторинг модуля**


## E. Метрики и реальные результаты

### METR: Рандомизированное контролируемое исследование (июль 2025)
- **Методология:** 16 опытных open-source разработчиков, 246 реальных задач на знакомых кодовых базах. Случайное назначение AI/не-AI условий.
- **Результат:** AI-assisted разработчики были **на 19% медленнее** в среднем.
- **Прогноз самих разработчиков:** Ожидали ускорение на 24%.
- **Причина:** Время на ревью вывода AI, составление промптов и ожидание ответов превысило экономию на реализации — для разработчиков, которые *уже хорошо знали* свои кодовые базы.
- **Обновление (февраль 2026):** После исправления методологии — **ускорение на 18%** с returning-разработчиками. Ключевой урок: измерить продуктивность AI методологически сложно.

### DORA Report 2025 (Google)
AI — «усилитель, увеличивающий существующие сильные и слабые стороны организации». Команды с сильными инженерными практиками → мультипликативная отдача. Команды со слабыми → мультипликативный технический долг.

### JetBrains Developer Ecosystem Survey 2025
- **85%** разработчиков регулярно используют AI-инструменты
- **66%** считают, что традиционные метрики продуктивности «сломаны» в эру AI

### Агрегированные данные (2025-2026)
- **41%** всего кода сейчас генерируется AI
- **Самоотчётный прирост:** 10-30% в зависимости от типа задачи
- **AI отлично работает для:** boilerplate, тесты, объяснение кода, документация, изолированные баги
- **AI плохо работает для:** архитектурные решения, кросс-модульные зависимости (auth, concurrency), доменная логика, код «который трогает всё»

### Render.com AI Coding Agents Benchmark (2025)
Тест Cursor, Claude Code, Gemini CLI, Codex на реальных задачах:
- **Cursor** — лучший средний балл (8/10), лидер по скорости настройки
- **Claude Code** — лучший UX, но проблемы с large-context
- **Gemini CLI** — лучший в large-context рефакторинге
- **Codex** — сильные модели, слабый DX
