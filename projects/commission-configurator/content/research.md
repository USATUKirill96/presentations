# Research: Конфигуратор комиссии — пошаговая инструкция для менеджеров

## Ключевые инсайты (топ-5 главных мыслей для презентации)

1. Конфигуратор позволяет менеджерам самостоятельно создавать и менять правила комиссии без привлечения разработчиков — это главная ценность инструмента.
2. Система построена на принципе неизменяемой истории: редактирование правила создаёт новую версию, а не переписывает старую, что защищает финансовые данные.
3. Приоритеты работают детерминировано: seller-specific > category L4 > category L1 > default — это исключает неоднозначность при выборе правила.
4. Preview (предпросмотр) — ключевой инструмент проверки: менеджер может увидеть, какое правило сработает для конкретной транзакции, ещё до того как оно вступит в силу.
5. Система имеет встроенные guardrails (защитные барьеры), которые не дадут создать противоречивые или «сломанные» правила — ошибки блокируются на этапе валидации.

## A. Основные понятия системы

### Правило (Rule Identity) и версия правила (Rule Version)

- Что это: Rule identity — это долгоживущий бизнес-объект, например «Дефолтное правило маркетплейса» или «Премиальное соглашение для Селлера 773». Rule version — одна неизменяемая сохранённая версия этого правила. Бизнес-изменение создаёт новую версию вместо перезаписи истории.
- Пример: Менеджер хочет поменять комиссию для селлера. Он не «редактирует» старое правило — он создаёт новую версию. Старая версия остаётся в истории с точными датами действия.
- Почему важно: Это защищает финансовые расчёты. Если позже нужен возврат — система берёт ту версию правила, которая действовала в момент заказа, а не текущую.

### Разница между дефолтным и кастомным правилом

- Что это: Default rule — обязательное базовое правило-фоллбек, которое применяется когда ни одно более специфичное правило не подходит. Custom rule — более узкое правило, которое перекрывает дефолтное поведение для конкретного бизнес-случая.
- Пример: Маркетплейс имеет одно дефолтное правило с комиссией 15%. Для конкретного крупного селлера создано кастомное правило с комиссией 10% — оно «побеждает» дефолтное для этого селлера.
- Почему важно: Маркетплейс всегда должен иметь ровно одно дефолтное правило (single default invariant). Нельзя создать второе дефолтное или удалить единственное — система это блокирует.

### Scope (область применения)

- Что это: Scope описывает ГДЕ правило будет работать. Включает: привязка к селлерам, категории товаров (L1/L4, включения/исключения), тип фулфилмента (FBO/DBS), VAT-контекст и ценовое окно (price window).
- Пример: Scope правила может быть «селлер 773 + FBO + все категории» — тогда это правило сработает только для товаров этого селлера, доставляемых со склада маркетплейса.
- Почему важно: Чем точнее scope, тем выше приоритет правила. Seller-specific rule всегда побеждает category rule, а category L4 побеждает L1.

### Fee Payload (результат)

- Что это: Fee payload описывает ЧТО происходит после того, как правило сработало — какие комиссии и сборы применяются к транзакции.
- Пример: Payload может включать: комиссию 12% (flat-threshold), fulfillment fee 50 руб., marketing fee 3%, return handling fee 30 руб.
- Почему важно: Кастомное правило (custom rule) должно быть «complete» — оно полностью автономно и не наследует поля из дефолтного правила. Если вы создаёте кастомное правило, нужно указать ВСЕ параметры.

### Жизненный цикл правила

- Что это: Каждая версия правила проходит стадии: DRAFT (черновик) → READY (готово, провалидировано) → ACTIVE (активно, в работе) → EXPIRED (истекло) → ARCHIVED (в архиве).
- Пример: Менеджер создаёт правило — оно в статусе DRAFT. Проходит валидацию — READY. Наступает дата effective_from — ACTIVE. Наступает effective_to — EXPIRED. Менеджер архивирует — ARCHIVED.
- Почему важно: Переходы между статусами строго контролируются. Submit переводит в бизнес-использование. Withdraw отзывает обратно (если разрешено). Archive убирает из будущего matching, но сохраняет историю.

## B. Создание и настройка правил

### UJ-01: Создание/изменение дефолтного правила маркетплейса

- Что это: Bootstrap the default fallback — создание или обновление единственного дефолтного правила маркетплейса.
- Пример: Маркетплейс запускается, нужна базовая комиссия. Менеджер создаёт дефолтное правило: scope = wildcard (без ограничений), payload = комиссия 15% + fulfillment fee 50 руб.
- Почему важно: Маркетплейс не может работать без дефолтного правила (single default invariant). Оно всегда является последним фоллбеком. Изменение дефолтного правила = создание новой версии.
- Ключевые термины: default rule, single default invariant, untargeted default, wildcard scope, new version.

### UJ-02: Создание правила для конкретного селлера (Seller-specific Rule)

- Что это: Создание кастомного правила с привязкой к одному или нескольким конкретным селлерам. Это наивысший приоритет.
- Пример: Селлер 773 договорился о специальных условиях: комиссия 8% вместо стандартных 15%. Менеджер создаёт seller-specific rule, привязывает селлера через seller assignment.
- Почему важно: Seller-specific rule имеет наивысший приоритет в системе. Можно настроить scope дополнительно: fulfillment applicability (FBO/DBS), VAT context, price window.
- Ключевые термины: custom rule, seller-specific rule, seller assignment, fulfillment applicability, VAT context, price window, complete custom rule.

### UJ-03: Создание правила по категориям с включениями/исключениями

- Что это: Category rule — правило, нацеленное через категории товаров вместо ID селлеров.
- Пример: Менеджер хочет комиссию 10% для категории «Электроника» (L1), но исключить подкатегорию «Смартфоны» (L4), для которой будет отдельное правило с комиссией 7%.
- Почему важно: Система поддерживает иерархическую структуру: included category L1, included category L4, excluded category L1, excluded category L4. Специфичность L4 > L1 — более узкая категория побеждает более широкую. Excluded категории убираются из scope перед финальным matching.
- Ключевые термины: category rule, included/excluded category L1/L4, category specificity, category ancestry.

### UJ-04: Редактирование существующего правила (создание новой версии)

- Что это: «Редактирование» в системе — это не изменение существующей записи, а создание новой версии правила (new version). Старая версия остаётся неизменной.
- Пример: Комиссия для селлера 773 менялась: версия 1 (10%), версия 2 (8%), версия 3 (12%). Каждая имеет свои effective_from/effective_to. Все хранятся в rule version history.
- Почему важно: Immutable history — раз версия повлияла на денежные расчёты, она не может быть переписана. Это основа финансовой безопасности.
- Ключевые термины: rule identity, rule version, immutable history, new version, rule version history.

### UJ-05: Деактивация и архивирование правила

- Что это: Остановка будущего использования правила. Два механизма: установка effective_to (дата окончания) и archive (полное удаление из будущего matching).
- Пример: Коммерческое соглашение с селлером 773 закончилось. Менеджер устанавливает effective_to на сегодня. С завтрашнего дня для этого селлера снова работает дефолтное правило (fallback).
- Почему важно: Future-only lifecycle change — изменения влияют ТОЛЬКО на будущее. Прошлые транзакции, рефанды, аудит — всё остаётся привязано к тому правилу, которое действовало в момент продажи.
- Ключевые термины: future-only lifecycle change, effective_to, future-facing archive, historical readability, fallback.

### Параметры scope

- **Fulfillment applicability (FBO/DBS):** Правило может применяться только к товарам, доставляемым со склада маркетплейса (FBO), только к товарам с доставкой селлера (DBS), или к обоим типам.
- **VAT context:** Правило может быть ограничено конкретным VAT-контекстом или работать в любом.
- **Price window:** Ценовое окно ограничивает применимость правила диапазоном цены товара.

## C. Настройка комиссий и сборов (Fee Payload)

### Типы комиссий: flat-threshold и tiered

- Что это: Flat-threshold commission — модель, где ставка меняется один раз вокруг одного порога. Tiered commission — модель с несколькими ценовыми полосами, где ставка различается для каждой полосы.
- Пример: Flat-threshold: до 1000 руб. — комиссия 15%, свыше 1000 руб. — 12%. Tiered: 0-500 руб. → 20%, 500-2000 руб. → 15%, 2000+ руб. → 10%.
- Почему важно: Выбор модели зависит от коммерческой стратегии. Flat-threshold проще, tiered позволяет более тонкую настройку под ценовые сегменты.

### Fulfillment fee: пороговый и фиксированный

- Что это: Threshold fulfillment fee — сбор за фулфилмент, который меняется в зависимости от порога. Fixed fulfillment fee — постоянная сумма вне зависимости от цены товара.
- Пример: Threshold: до 500 руб. — 80 руб., свыше 500 руб. — 50 руб. Fixed: всегда 60 руб. за единицу.
- Почему важно: Fulfillment fee обязателен для FBO-товаров. Правильный выбор модели влияет на экономику логистики.

### Marketing fee и Return handling fee

- Что это: Marketing fee — дополнительный маркетинговый сбор (процент от цены). Return handling fee — фиксированная сумма за обработку возврата.
- Пример: Marketing fee: 3% от цены товара. Return handling fee: 30 руб. за каждый обработанный возврат.
- Почему важно: Эти сборы опциональны в рамках правила, но если кастомное правило — complete custom rule, нужно указать все параметры явно (нельзя «наследовать» из дефолтного правила).

### Base price mode: с НДС или без НДС

- Что это: Определяет, на какую базовую цену рассчитывается комиссия — цену inclusive of VAT (с НДС) или exclusive of VAT (без НДС).
- Пример: Товар стоит 1200 руб. с НДС (1000 руб. без). При base price mode = inclusive комиссия 10% = 120 руб. При exclusive = 100 руб.
- Почему важно: Неправильный выбор base price mode может привести к системной ошибке в расчётах комиссии по всем транзакциям, попадающим под правило.

### Complete custom rule

- Что это: Кастомное правило, которое работает полностью автономно. Оно не наследует никакие поля из дефолтного правила — все параметры (commission, fulfillment fee, marketing fee, return handling fee, base price mode) должны быть указаны явно.
- Пример: Создавая правило для селлера 773, менеджер должен заполнить ВСЕ поля: тип комиссии, ставки, fulfillment fee, marketing fee, return handling fee, base price mode.
- Почему важно: Это design decision системы (DR-04). Частичное наследование из дефолтного правила не поддерживается. Это исключает неоднозначность: глядя на любое кастомное правило, менеджер видит полную картину без необходимости «складывать» его с дефолтным.

## D. Проверка и приоритеты

### Порядок приоритетов

- Что это: Детерминированный порядок, по которому система решает, какое правило «побеждает» для конкретной транзакции: seller-specific > category L4 > category L1 > default.
- Пример: Селлер 773 продаёт товар в категории «Электроника» (L1), подкатегория «Смартфоны» (L4). Есть правила для каждого уровня. Побеждает seller-specific rule (если есть), иначе L4, иначе L1, иначе default.
- Почему важно: Порядок жёсткий и не допускает неоднозначности. Система НИКОГДА не «угадывает» между двумя правилами одного приоритета — если такая ситуация возникает, это same-priority overlap и оно блокируется на этапе валидации.

### Preview (предпросмотр)

- Что это: Механизм, позволяющий заранее проверить, какое правило сработает для конкретного набора параметров (селлер, категория, цена, fulfillment, VAT, дата).
- Пример: Менеджер хочет убедиться, что новое правило для селлера 773 действительно «победит». Он вводит в preview: seller=773, category=Electronics/Smartphones, price=15000, date=завтра. Система показывает: selected rule = «Правило для селлера 773 v2».
- Почему важно: Preview позволяет проверить корректность ДО того, как правило повлияет на реальные транзакции. Это главный инструмент верификации для менеджера.

### Как читать результат: selected rule, matched basis, selection reason

- Что это: Ответ системы на preview или аудит-запрос. Selected rule — какое правило выиграло. Matched basis — по каким бизнес-параметрам оно совпало. Selection reason — человекочитаемое объяснение почему.
- Пример: Selected rule: «Seller 773 Premium v2». Matched basis: seller_id=773, fulfillment=FBO. Selection reason: «Seller-specific rule matched by seller assignment, priority: seller > category > default».
- Почему важно: Это даёт прозрачность и объяснимость. Если финансы или саппорт спрашивают «почему такая комиссия?» — можно показать конкретный ответ системы с обоснованием.

### Исторический lookup

- Что это: Возможность узнать, какое правило действовало для конкретной прошлой транзакции. Отвечает на вопрос «what did happen?» (в отличие от preview, который отвечает «what would happen?»).
- Пример: Поступил запрос на возврат за заказ от 15 марта. Исторический lookup показывает: в тот момент действовало правило «Seller 773 Premium v1» с комиссией 10%. Рефанд рассчитывается по этим данным.
- Почему важно: Refund-safe replay — возвраты ВСЕГДА рассчитываются по сохранённому прошлому решению, а не по текущим правилам.

### Ошибки валидации

- Что это: Система блокирует проблемные конфигурации ДО того, как правило войдёт в бизнес-использование.
- Пример ошибок:
  - **Same-priority overlap:** Два правила одного приоритета покрывают одну и ту же бизнес-ситуацию. Система не знает, кого выбрать → ошибка.
  - **Duplicate business meaning:** Новое правило создаёт тот же смысл, что уже существующее в том же временном окне → ошибка.
  - **Category scope empty after exclusions:** Все включённые категории исключены — правило никогда не сработает → ошибка.
  - **Category path unresolved:** Система не может определить путь категории → ошибка.
- Почему важно: Guardrails защищают от человеческих ошибок. Лучше получить ошибку валидации на этапе создания, чем обнаружить проблему в продакшене.

## E. Безопасность и гарантии

### Immutable history (неизменяемая история)

- Что это: Как только версия правила повлияла на денежные расчёты, она не может быть изменена или удалена. Hard delete запрещён.
- Пример: Версия 1 правила для селлера действовала с 1 по 15 марта. За это время прошло 1000 транзакций. Даже если менеджер создаст версию 2 — версия 1 останется навсегда с привязкой к тем 1000 транзакциям.
- Почему важно: Это основа финансовой целостности системы. Аудит, бухгалтерия, финансовая отчётность — все зависят от того, что прошлое не может быть переписано.

### Refund-safe replay

- Что это: При обработке возвратов система использует СОХРАНЁННОЕ прошлое решение (historical decision), а не перерешивает по текущим правилам.
- Пример: Заказ от 1 марта, тогда действовала комиссия 10%. Сейчас комиссия 15%. При возврате 20 апреля — рефанд считается по ставке 10% (которая была на момент заказа), а не по текущей 15%.
- Почему важно: Без этого принципа любое изменение комиссии ретроактивно влияло бы на все рефанды, создавая финансовый хаос.

### Guardrails (защитные механизмы)

- Что это: Встроенные правила системы, которые блокируют опасные или неоднозначные конфигурации.
- Примеры:
  - **Single default invariant:** Всегда ровно одно дефолтное правило. Нельзя ноль, нельзя два.
  - **Untargeted default:** Дефолтное правило не может быть нацелено на конкретных селлеров или категории — оно должно оставаться глобальным.
  - **Same-priority overlap:** Запрет на два правила одного приоритета, покрывающих одну ситуацию.
  - **Duplicate business meaning:** Запрет на дублирование смысла существующего правила.
  - **Category scope empty after exclusions:** Блокировка «пустых» правил.
- Почему важно: Guardrails делают систему «защищённой от дурака». Менеджер не может случайно сломать расчёт комиссий — система остановит его и объяснит что не так.

### Future-only lifecycle change

- Что это: Любое изменение жизненного цикла (архивирование, установка effective_to) влияет ТОЛЬКО на будущие транзакции. Прошлые решения не затрагиваются.
- Пример: Менеджер архивирует правило для селлера 773 сегодня. Завтра для этого селлера сработает fallback (дефолтное правило). Но все прошлые транзакции этого селлера по-прежнему привязаны к архивированному правилу.
- Почему важно: Это гарантирует, что операционные действия менеджера (архивирование, деактивация) не имеют ретроактивных последствий. Можно действовать уверенно — прошлое защищено.
