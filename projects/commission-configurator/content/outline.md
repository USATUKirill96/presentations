# Структура презентации: Конфигуратор комиссии

## Метаданные
- Название: Как пользоваться конфигуратором комиссии
- Подзаголовок: Настройка правил комиссии: от создания правила до проверки результата
- Автор: 
- Слайдов: 26
- Продолжительность: ~15 мин (самостоятельное чтение)

## Нарратив (одним абзацем)

Презентация ведёт менеджера от вопроса «к кому обращаться, чтобы поменять комиссию?» к ответу «теперь ты можешь сделать это сам». Мы начинаем с простого объяснения базовых понятий, затем постепенно погружаемся в конкретные сценарии: создание правил, настройка комиссий, проверка через preview. Кульминация — полный цикл от создания правила до подтверждения результата. Финал — приглашение попробовать самостоятельно в тестовой среде.

### Слайд 5: [Тип: concept]
**Заголовок:** Правило и версия правила
**Ключевое сообщение:** Правило — это долгоживущий объект. Изменения создают новые версии, не затирая старые.
**Контент (тезисы):**
- Rule identity = бизнес-объект («Дефолтное правило», «Соглашение Селлера 773»)
- Rule version = конкретная сохранённая версия с датами действия
- «Редактирование» = создание новой версии
- Старые версии остаются навсегда
**Цель:** Объяснить ключевую архитектурную идею — immutable versioning

### Слайд 6: [Тип: concept]
**Заголовок:** Дефолтное правило vs Кастомное правило
**Ключевое сообщение:** Одно обязательное + много специальных
**Контент (тезисы):**
- Default rule = всегда одно, фоллбек на все случаи
- Custom rule = перекрывает дефолтное для конкретного случая
- Default нельзя удалить, нельзя создать второе
- Custom = полностью автономное (complete custom rule)
**Цель:** Дать базовое понимание двух типов правил

### Слайд 7: [Тип: concept]
**Заголовок:** Scope — где правило работает
**Ключевое сообщение:** Scope определяет, к каким транзакциям применяется правило
**Контент (тезисы):**
- Привязка к селлерам (seller assignment)
- Категории товаров (L1/L4, включения/исключения)
- Тип фулфилмента (FBO / DBS / оба)
- VAT-контекст
- Ценовое окно (price window)
**Цель:** Показать все «рычаги» настройки области применения

### Слайд 8: [Тип: concept]
**Заголовок:** Fee Payload — что применяется
**Ключевое сообщение:** Payload = набор комиссий и сборов, которые «включатся» когда правило сработает
**Контент (тезисы):**
- Комиссия (commission): flat-threshold или tiered
- Fulfillment fee: пороговый или фиксированный
- Marketing fee
- Return handling fee
- Base price mode: с НДС или без
**Цель:** Дать обзор всех финансовых параметров правила

### Слайд 9: [Тип: diagram]
**Заголовок:** Жизненный цикл правила
**Ключевое сообщение:** Каждая версия проходит 5 стадий
**Контент (тезисы):**
- DRAFT → READY → ACTIVE → EXPIRED → ARCHIVED
- Submit: переход в бизнес-использование
- Withdraw: отзыв (если ещё не активировано)
- Archive: убрать из будущего matching
**Цель:** Визуализировать жизненный цикл, чтобы менеджер знал где «живёт» его правило

### Слайд 18: [Тип: concept]
**Заголовок:** Типы комиссий: flat-threshold vs tiered
**Ключевое сообщение:** Два способа посчитать основную комиссию
**Контент (тезисы):**
- Flat-threshold: одна ставка меняется у одного порога (до 1000₽ → 15%, свыше → 12%)
- Tiered: несколько полос (0-500 → 20%, 500-2000 → 15%, 2000+ → 10%)
- Flat-threshold проще, tiered гибче
- Выбор зависит от коммерческой стратегии
**Цель:** Объяснить разницу между моделями комиссий

**Контент (тезисы):**
- Fulfillment fee: threshold (меняется от цены) или fixed (постоянный)
- Marketing fee: процент от базовой цены
- Return handling fee: фиксированная сумма за возврат
- Все поля обязательны для complete custom rule

### Слайд 20: [Тип: warning]
**Заголовок:** Base price mode — не перепутай!
**Ключевое сообщение:** С НДС и без НДС — разница в расчётах может быть существенной
**Контент (тезисы):**
- Inclusive of VAT: комиссия считается от цены С НДС
- Exclusive of VAT: от цены БЕЗ НДС
- Пример: товар 1200₽ (с НДС) / 1000₽ (без). Комиссия 10%: 120₽ vs 100₽
- Выбирай осознанно — ошибка влияет на ВСЕ транзакции правила
**Цель:** Предупредить о частой ошибке, дать конкретный пример

**Тизер:** Почему можно действовать уверенно

### Слайд 26: [Тип: concept]
**Заголовок:** Система защищает от ошибок
**Ключевое сообщение:** Guardrails не дадут сломать расчёты
**Контент (тезисы):**
- Immutable history: прошлое нельзя переписать
- Refund-safe replay: возвраты по историческим данным
- Single default invariant: всегда ровно одно дефолтное правило
- Same-priority overlap: система не допустит конфликтов
- Future-only changes: архивирование не затрагивает прошлое
**Цель:** Дать уверенность, что инструмент безопасен

### Слайд 27: [Тип: warning]
**Заголовок:** Частые ошибки и как их избежать
**Ключевое сообщение:** Знай типичные проблемы — решай их до submit
**Контент (тезисы):**
- Same-priority overlap → сузить scope одного из правил
- Duplicate business meaning → объединить в одно правило
- Category scope empty → проверить что excluded не покрывают все included
- Забыли заполнить поле → complete custom rule требует ВСЕ поля
**Цель:** Предупредить о граблях, дать решение для каждой ситуации

- Приоритеты: seller > L4 > L1 > default

- Всегда проверяй через Preview после изменений
- Archive/deactivate влияют только на будущее

## Переходы между разделами

- Введение → Понятия: «Прежде чем создавать правила, давайте разберёмся в 5 ключевых терминах»
- Понятия → Создание: «Теперь, когда вы знаете словарь, переходим к действиям»
- Создание → Комиссии: «Мы научились создавать правила — теперь разберёмся что именно туда вписывать»
- Комиссии → Проверка: «Правило настроено — как убедиться что всё сработает правильно?»
- Проверка → Безопасность: «А что если я ошибусь? Система вас защищает.»
- Безопасность → Заключение: «Всё безопасно, всё объяснимо — пора попробовать самому»
