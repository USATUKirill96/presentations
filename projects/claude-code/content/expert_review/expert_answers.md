# Expert Answers to Critic's Questions

### Q1
- Status: partially confirmed
- Short answer: the critic is right. `85%` is reliably confirmed with a clear sample; `41%` in its current form I could not reliably confirm and recommend removing or strictly tying to a specific study if such a primary source is found.
- What the source says: JetBrains states that the 2025 study is based on `24,534 developers across 194 countries` and that `85% of developers regularly use AI tools for coding and development`. For `41% of code is already generated by AI`, there is no comparable primary source in the current set; as stated, this is too broad and fragile a number.
- Recommended edit in presentation: on slide 3, keep `85% of developers regularly use AI tools` with a footnote to JetBrains, and replace `41%` with a qualitative statement like `AI has already become part of daily development, but quality and effectiveness heavily depend on context and process`.
- Sources:
  - https://blog.jetbrains.com/research/2025/10/state-of-developer-ecosystem-2025/

### Q2
- Status: partially confirmed
- Short answer: this is not a mass norm, but an already achievable advanced workflow. Presenting it as `well-configured teams can already work this way`, rather than as a typical current state of the market, would be more honest.
- What the source says: GitHub Copilot coding agent officially has `work on an issue` and `create a pull request` scenarios, and MCP allows expanding access to external data and tools. ### Q3
- Status: confirmed
- Short answer: the critic is right. Here similar but not identical mechanisms are indeed conflated. They are united by the function of `project-level instructions`, but not by format, scope, or resolution logic.
- What the source says: Anthropic writes that Claude Code reads `CLAUDE.md` at the start of each session and separately clarifies that `Claude Code reads CLAUDE.md, not AGENTS.md`, although `AGENTS.md` can be imported. GitHub Copilot distinguishes between `.github/copilot-instructions.md`, path-specific `.github/instructions/**/*.instructions.md`, and agent instructions in `AGENTS.md` / `CLAUDE.md` / `GEMINI.md`.
- Recommended edit in presentation: keep the thesis `one idea, different implementations`, but add a short framing: `This is not a single standard, but a family of similar repo-level instruction mechanisms`.
- Sources:
  - https://docs.anthropic.com/en/docs/claude-code/memory

### Q4
- Status: confirmed
- Short answer: the critic is right. The idea of guardrails and deny rules is strong, but specific configs are not universal and not always painlessly portable between tools.
- What the source says: Claude Code hooks actually allow blocking tool calls via `PreToolUse` and `permissionDecision: "deny"`, including for destructive commands like `rm -rf`. MCP security docs separately emphasize that external servers need strict security controls and per-client consent. This confirms the guardrails idea, but not the thesis `enable the same config everywhere`.
- Recommended edit in presentation: add the caveat `configurations are illustrative; check event support, permissions, and scope for your tool`. On slide 9, better to separate `hooks` and `rules` so a Cursor rule is not presented as a deterministic hook.
- Sources:
  - https://docs.anthropic.com/en/docs/claude-code/hooks
  - https://modelcontextprotocol.io/specification/2025-06-18/basic/security_best_practices
  - https://docs.cursor.com/context/rules-for-ai

### Q5
- Status: partially confirmed
- Short answer: as a universal default for non-trivial tasks, this workflow is defensible; as a mandatory process for everything — it is not. It should be presented as a recommended pattern with explicit applicability boundaries.
- What the source says: Cursor and Claude Code both support planning/read-only workflows: Cursor has `Ask` as a read-only mode for exploration and a separate `Plan Mode`; Claude Code and the tooling ecosystem around it have explicit separation of research and execution. But none of these sources claim that all tasks must go through a full 4-phase cycle.
- Recommended edit in presentation: add an applicability criterion: `Use the full cycle for tasks with non-obvious architecture, multiple files, external integrations, or high error risk; for typos, imports, and local fixes the cycle can be shortened`.
- Sources:
  - https://cursor.com/changelog/0-48-x
  - https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents

### Q6
- Status: confirmed
- Short answer: the critic is right. A more precise formulation is not `AI always writes tests and implementation simultaneously`, but `the agent by default optimizes for task completion unless explicitly constrained`.
- What the source says: this aligns with the overall picture from Anthropic about the importance of explicit context/rules and from METR about verification overhead: without constraints, the agent tends to quickly produce a working-looking result rather than follow a specific engineering discipline like Red-first TDD.
